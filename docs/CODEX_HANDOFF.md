# Codex 交接指南

这份文档给新开的 Codex 窗口先读，用来快速恢复 LexScope Agent 的项目上下文和维护习惯。

## 项目定位

LexScope Agent 是一个面向中文民商事法律场景的智能问答系统。核心体验是：用户上传 PDF/TXT 法律资料，直接提问，系统用 Spring AI、检索增强、引用来源、会话版本和前端聊天界面给出可追溯回答。

当前前端体验优先级：
- 像聊天产品一样直接可用，少放说明书式文字。
- 默认演示入口应能直接上传资料和提问。
- 普通用户界面只展示必要信息；复杂配置放到设置或高级区域。
- 不删除后端已有治理能力，只在必要时简化前端展示。

## 新窗口开始方式

每次开始处理需求前：
- 先查看 `git status --short`，确认已有未提交改动。
- 不覆盖用户已有改动；如果同一个文件已有改动，要在当前内容上继续小范围修改。
- 优先阅读相关实现文件，再动手修改。
- 涉及前端体验时，保持现有 LexScope 聊天风格：简洁、可用、少解释。
- 涉及 8088 可见的前端改动时，完成验证后重建 `web` 容器，让用户刷新 `http://localhost:8088` 就能看到效果；纯文档或后端内部改动不用重建。

可直接使用的提示词：

```text
请先阅读 docs/CODEX_HANDOFF.md，再处理我的需求。

注意：
1. 不要覆盖工作区已有未提交改动。
2. 每次项目改进后都要同步更新 ARCHITECTURE.md、CHANGELOG.md、INTERVIEW_NOTES.md。
3. 最终回复必须输出变更报告：目标、改动文件、具体改动、实现效果、验证方式、风险与后续。
4. 按 docs/CODEX_HANDOFF.md 里的 Git 策略检查改动范围；没有我明确要求，不要自动提交。
5. 如果涉及前端体验，请优先保持现在的 LexScope 聊天产品风格：少说明、多直接可用。
6. 完成前如果影响 8088 页面效果，要重建 web 容器让我刷新后能看到。
```

## 文档同步规则

每次做项目改进或修改代码后，必须同步更新这三份项目记录文档：
- `ARCHITECTURE.md`：记录架构、数据流、前后端职责、关键交互变化。
- `CHANGELOG.md`：在 `[未发布]` 下追加简短变更记录。
- `INTERVIEW_NOTES.md`：把能用于面试表达的排查过程、优化效果和项目亮点沉淀下来。
- 这些随项目持续更新的维护文档默认使用中文；必要的技术名词、命令、路径、接口名和 commit message 格式可以保留英文。

每次完成改进后，还要在最终回复里输出一份简短变更报告，固定包含：
- 目标：这次要解决什么问题。
- 改动文件：列出主要文件。
- 具体改动：说明做了哪些行为变化。
- 实现效果：用户现在能看到或感受到什么变化。
- 验证方式：说明跑过哪些检查，或哪些检查未跑。
- 风险与后续：说明仍可能存在的风险、缓存、重启、手动验证事项。

如果改动只是纯文案或很小的样式微调，三份文档可以简短记录；如果涉及接口、鉴权、上传、流式输出、会话、成本治理或部署代理，必须写清楚原因、处理方式和效果。

## Git 提交策略

每次提交前固定执行三步：
- 查看改动范围：确认 `git status --short` 和 diff 中只有本次任务相关内容。
- 运行必要验证：至少跑对应前端或后端的类型检查、构建或测试。
- 确认文档同步：检查 `ARCHITECTURE.md`、`CHANGELOG.md`、`INTERVIEW_NOTES.md` 是否已按本次改动更新。

提交约定：
- 不自动提交，除非用户明确说“提交”或“帮我 commit”。
- 提交前先说明将包含哪些文件。
- 一个 commit 只放同一任务相关内容，不混入无关改动。
- 提交信息使用 `type(scope): short summary`。
- 示例：`fix(frontend): stabilize streaming chat`。
- 示例：`docs(process): add codex handoff guide`。
- 如果代码修复和流程文档能独立回滚，可以拆成两个 commit；否则用一个清晰 commit。

## 常用验证

前端改动优先验证：

```powershell
cd frontend
npm run typecheck
npm run build
```

如果这次改动会影响 8088 页面效果，验证通过后回到项目根目录重建前端容器：

```powershell
docker compose up -d --build web
```

后端改动优先验证：

```powershell
mvn test
```

本地演示入口：
- 前端控制台：`http://localhost:8088`
- 后端 API：`http://localhost:8080`
- Swagger UI：`http://localhost:8080/swagger-ui/index.html`
