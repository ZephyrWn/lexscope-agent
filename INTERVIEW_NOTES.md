# LexScope Agent 面试笔记

## 2026-07-01 上传问题修复表达

今天排查到上传失败不是按钮交互问题，而是浏览器里残留的旧身份信息让后端返回 403。我没有改后端接口，而是在 8088 的上传代理入口做了兜底：上传 PDF 时统一使用本地演示上传配置，并清掉旧的 Authorization。这样演示时用户打开页面就能直接上传资料，同时原来的鉴权、入库和问答链路仍然保留。

后续继续排查发现，浏览器 multipart 上传还会自动带 Origin，后端把 8088 代理入口误判成跨域请求。我在上传专用代理里清掉 Origin，让页面上传和命令行上传走到同一条后端链路，最终解决了“命令行能上传、页面不能上传”的差异。

资料导入后不能输出的原因也类似：聊天流式接口仍然被浏览器带来的 Origin 或旧 Authorization 拦住。我把问答接口也纳入 8088 的演示代理兜底，保证上传后可以直接提问；同时把输入框改成随内容自动增高，让用户输入较长案情或问题时更自然。

长回答页面里还优化了布局滚动：左侧记录栏固定在屏幕内，右侧聊天区独立滚动。这样用户查看长答案时，记录、用量和设置入口不会被滚走，整体更像成熟聊天产品。

我还把答案顶部的一排固定功能标签去掉，改成每次回答结束后给用户几个自然的延伸追问建议。这样页面不再像功能说明书，而是更像 GPT 式对话：用户看完答案后能顺着证据、风险、诉讼或合同条款继续问下去。同时增加了拖拽上传资料，用户把 PDF 拖到页面就能进入同一条上传链路，降低首次使用成本。

继续优化流式输出时，我把问题分成两层处理：第一层是让生产 Nginx 代理和本地 Vite 代理的演示鉴权行为一致，避免 8088 能用但开发模式仍然被旧 Authorization 或 Origin 拦截；第二层是增强前端 SSE 解析和错误展示，让页面能保留后端真实错误，而不是所有失败都归类成“服务配置错误”。这样下次遇到首轮失败、重试成功的情况，可以直接从页面详情判断是权限、CORS、预算保护、模型超时还是流式连接问题。

界面上，我把左侧“本月用量 / 预算”简化为只显示本月用量，保留后端预算治理但不把默认预算额度展示给普通用户；同时把“编辑后重发”的输入框加高，让长问题编辑更像正式文本输入，而不是挤在一个小框里。

后来继续顺着聊天产品体验做了微调：按钮文案从“编辑后重发”收敛成“编辑”，但底层仍然保留创建新版本的安全逻辑；编辑框横向加宽，方便修改长问题。Agent 处理过程改为默认折叠，只露出步骤数和耗时摘要，需要排查时再展开完整 trace，这样普通对话更干净，可追溯能力也没有丢。

分组和归档问题的根因在于两个状态层没有对齐：分组下拉原来只从会话记录里反推，所以没有记录的空分组会消失；归档同步原来直接调用点更新接口，本地新建记录还没上云时后端找不到记录。我把分组本身作为前端本地状态持久化，同时把归档/置顶同步改成保存完整会话快照，这样既不新增后端分组表，也能让本地新记录第一次操作时自动 upsert 到云端。

在会话同步体验上，我把用户需要理解的“从云端读取 / 保存至云端”降级成兜底操作，主流程改成自动同步。实现上没有新增后端接口，而是在前端保留 localStorage 即时保存，再用防抖队列把会话内容、分支和模型配置通过已有 upsert 接口静默保存到云端；流式回答时先不打断，等最终答案、引用和 trace 都落定后再保存快照。首次打开且本地没有缓存时才静默拉云端，避免自动读取覆盖本机还没同步的记录。这个点面试时可以表达为：我把“用户手动管理数据”改成“系统默默保证数据安全”，同时保留手动按钮作为异常兜底。

PDF/RAG 问答展示也做了一次“从工程调试态到正式产品态”的收敛。原来答案正文、引用区域和相关依据会重复出现 `source=...`、`chunk=...` 这类内部字段，看起来像把向量检索调试结果直接暴露给用户。现在后端保留调试信息但放进 `debug`，前端默认只展示“参考来源”卡片：编号、文件名、页码和简短摘要；正文只允许使用 `[1]`、`[2]` 这类编号引用，不再展示 chunk_index。

后续看到引用卡片里仍有 `tenant_id`、`chunk_index`、`distance`，我判断不是租户字段本身没用，而是后端文档内容格式化时把 metadata 拼进了摘要文本。处理方式是保留租户隔离和 debug 字段，但在生成上下文、证据摘要和前端卡片展示前做清洗；同时去掉固定的“资料匹配 40/25/20/15”条，避免没有真实统计时给用户一种伪动态分析的感觉。最后把聊天正文和引用卡片字号调大一点，让长法律文本更适合阅读。

参考来源点击返回 `missing or invalid credentials` 的原因不是后端接口坏了，而是前端用 `window.open` 直接打开受保护的 PDF 文件接口，新窗口不会带当前 API 客户端里的鉴权头。我保留后端安全规则，把点击动作改为前端受控请求：使用和聊天接口一致的 `authContext()` 携带 `X-API-Key`/租户头获取 PDF blob，再在页面内弹出来源预览。这样用户看到的是文件、页码和片段，而不是裸 JSON 错误。

为了解决新开 Codex 窗口容易丢项目习惯的问题，我新增了 `docs/CODEX_HANDOFF.md`：要求先看工作区状态、不要覆盖未提交改动、每次改进后同步三份文档，并固定 Git 提交策略和变更报告格式。这个做法相当于给项目增加一份轻量的“维护协议”，能减少上下文切换造成的遗漏。

我还把会随着项目持续更新的维护文档统一成中文表达，包括 Codex 交接指南、文档首页和变更日志。这样后续每次改动留下来的记录更适合自己快速复盘，也更容易直接转化成面试讲述；命令、接口名、路径和 commit message 格式仍保留英文，避免技术信息失真。

本文档记录 LexScope Agent 适合在简历和面试中复述的项目亮点、难点、优化点和效果。

## 1. 一句话介绍

LexScope Agent 是一个面向中文法律场景的智能问答系统，用户上传合同、判决书或法规资料后，可以直接提问，系统会基于资料内容生成带引用依据的法律分析。

## 2. 可以直接复述的项目表达

1. 我把一个偏工程化的 Agent / RAG 控制台，改造成了普通法律用户更容易理解的“法律智能问答”产品。
2. 用户不需要理解 JWT、向量检索、多租户这些技术概念，只需要按“上传资料、输入问题、查看分析”三个步骤使用。
3. 后端仍然保留鉴权、检索、引用溯源、流式回答、效果评测等能力，但前端把它们包装成更符合业务场景的功能。
4. 回答结果会按问题概括、争议焦点、相关依据、引用来源和风险提示来组织，更适合法律场景阅读。
5. 我还用 Docker Compose 把前端、后端、数据库和中间件串起来，保证项目能在本地一键启动和演示。

## 3. 项目亮点

### 法律场景产品化

优化前：前端更像开发者控制台，直接暴露 Agent、RAG、JWT、Tenant、SSE、检索评测等词，普通用户学习成本高。

优化后：前端主概念改为“法律智能问答”，菜单和首页引导围绕法律资料、法律问题和分析报告组织。

可以强调：我不仅实现了技术链路，也做了面向真实用户的产品化包装。

### 有依据的问答

系统不是只把问题丢给大模型，而是先从用户上传资料中检索相关片段，再结合这些片段生成回答，并展示引用来源。

可以强调：这样能降低模型胡编的风险，也方便用户回到原始材料核对。

### 异步资料入库

上传 PDF 后，系统会解析、切片、生成向量并写入知识库。入库可以走队列，避免大文件处理阻塞用户请求。

可以强调：这体现了工程上的稳定性和扩展意识。

### 会话和历史版本

用户可以保留问答记录，也可以把一次分析另存为历史版本、对比上一版或采用当前版本。

可以强调：法律分析经常需要多轮修改和对比，这个设计更贴合实际工作流。

### 效果评测

开发者可以用效果评测测试检索是否命中、引用是否覆盖、回答是否可靠。

可以强调：我不仅关注“功能能跑”，也关注“回答质量能不能被验证”。

## 4. 技术难点

### 前端产品化和后端能力解耦

难点：后端接口里有很多工程概念，但普通用户界面不应该直接展示这些词。

做法：不改后端接口和核心逻辑，只在前端信息架构、文案和默认展示上做分层，把高级配置收进设置和开发者工具。

### 流式回答和鉴权

难点：流式接口需要持续返回内容，同时还要经过安全校验，容易出现异步响应二次鉴权等问题。

做法：保留后端流式链路，前端统一展示为“实时输出”，把 SSE 等技术词隐藏起来。

优化点：发现有些答案虽然走了 SSE 接口，但后端把完整答案一次性作为一个 token 发出，用户看起来不像流式输出；我把完整答案切成小段逐步推送，并调整 Nginx 代理避免缓冲。

### 法律回答结构化

难点：普通 AI 回答往往是一整段文字，不适合法律场景快速阅读和核对。

做法：前端把回答结果按问题概括、争议焦点、相关依据、参考案例、初步分析、引用来源、风险提示组织。

### 简单问题快答

难点：原来简单问题也会走 ReAct 规划、检索和总结，等待时间偏长。

做法：增加保守的快答判断，普通解释类问题直接用一次模型调用回答；只要用户要求文档、引用、法条、案例、报告等依据型内容，就继续走完整研判链路。

## 5. 最近优化效果

- 修复真实 PDF 上传容易被前端代理默认大小限制拦截的问题，把 Nginx 上传上限和后端统一到 100MB，并把上传错误提示从“配置错误”改为按原因说明。
- 进一步把“上传资料所需的应用鉴权”和“模型 API Key 设置”拆开，上传入口固定使用内置上传密钥，避免用户设置里的旧凭证导致 403。
- 修复默认配置下上传资料被旧浏览器缓存 API Key 阻塞的问题：前端启动优先使用内置演示密钥，上传遇到鉴权错误会自动恢复默认配置并重试。
- 删除设置面板底部的“关于系统 / 使用说明”说明卡，把说明文字从普通设置路径里拿掉，减少用户打开设置时的阅读压力。
- 放大引导页上传卡片和按钮，并去掉底部 3 个快捷问题前的小图标，让首屏更简洁、更聚焦。
- 按参考图把引导页进一步做成聊天产品空状态：中间是一个上传资料卡片，下面用分隔线连接“直接输入问题”，再放 3 个常用问题按钮。
- 把本月用量从设置面板挪到左侧设置按钮上方，用户不用进入复杂设置，也能看到当前用量和预算。
- 将左侧用量卡片进一步简化为只显示本月用量，避免普通用户看到预算额度产生误解；后端成本治理能力仍保留。
- 修复本地开发代理和生产代理行为不一致导致的流式输出失败隐患，并增强 SSE 错误详情展示。
- 将 PDF/RAG 引用从 `source/chunk` 调试文本改成正式“参考来源”卡片，减少重复引用和内部字段外露。
- 进一步清洗 PDF/RAG 引用摘要里的 metadata，把 `tenant_id`、`chunk_index`、`distance` 等字段保留在后台 debug 中但不展示给普通用户；同时移除固定资料匹配百分比条，放大聊天和引用卡片字号。
- 修复参考来源卡片点击后的 PDF 预览鉴权问题，把裸打开后端文件 URL 改成前端携带鉴权头获取 PDF blob 并弹窗展示片段和页码。
- 将持续更新的维护文档统一调整为中文表达，降低后续复盘和面试准备成本。
- 新增 Codex 交接文档，把三份文档同步规则、固定变更报告和 Git 提交策略固定下来，减少后续维护时漏写记录或混入无关改动。
- 把法律智能问答首屏从“三步说明卡片”改成更像聊天产品的轻量入口：用户只看到“上传法律资料”或“直接提问”两个方向，不需要先理解一堆功能说明。
- 快捷问题从 4 个收敛到 3 个高频场景：合同风险、争议焦点、生成法律分析报告；结果模块标签不再一进页面就展示，等用户真正提问后再出现。
- 首屏上传按钮复用已有资料上传接口，PDF 可以直接上传；TXT 会读取到输入框辅助提问，Word 先用友好提示引导另存 PDF，避免普通用户看到接口或技术错误。
- 首页标题和菜单更像一个给普通用户用的法律问答产品，而不是工程控制台。
- 进一步删掉左侧重复的四宫格菜单，并把历史版本移入设置，让侧栏只保留新建问答和记录搜索，主操作集中到聊天输入区。
- 左侧记录区只保留搜索框，把分组筛选、归档显示和云端同步收到设置里，减少普通用户第一眼看到的管理按钮。
- 左侧栏改成更接近 ChatGPT 的使用方式：先搜索记录，再新建问答，下面是我的记录；每次新建都会生成“新对话”并出现在列表顶端。
- 放大左侧记录搜索框并统一聚焦态，让搜索入口更明显，也和新建问答按钮保持同一套视觉节奏。
- 修复实时输出的体验问题，让走 SSE 的回答真正分段出现在聊天框里，而不是等后端生成完后整段出现。
- 为简单法律解释问题增加快答路径，减少不必要的规划和检索步骤，让普通提问响应更快。
- 历史问答卡片隐藏 sessionId、chatId、uuid 这类内部标识，只展示标题和更新时间，让普通用户看到的是业务信息而不是技术字段。
- 去掉“我的记录”外层边框，只保留每条会话自己的卡片边界，让侧栏层级更轻，不再出现卡片套卡片。
- 历史版本的另存、对比和采用能力仍然保留，只是从侧栏移到设置的高级设置里，避免高级操作长期占据主界面。
- 主内容头部只保留主题切换，把分组、归档等低频动作收到设置里；清空当前会话不在普通设置面板展示，避免用户误触清空内容。
- 默认使用本地演示模型 API Key，避免用户首次打开时先理解“连接、登录、刷新令牌”等概念，设置里仍保留模型 API Key 入口，方便演示或替换服务。
- 设置面板改成“基础设置、数据管理、高级设置、关于系统”的结构，普通用户默认只看到模型 API Key 和说明，高级功能折叠保留。
- 云端同步、历史版本、分组管理、效果评测和用量统计都没有删除，只是放进折叠区，避免普通用户一打开设置就看到复杂选项。
- 移除输入框下方重复的快捷提示按钮，让问答区更像一个干净的聊天输入界面，用户可以直接输入自己的法律问题。
- 设置项和开发者工具被收起，用户打开页面后不会被 API Key、JWT、Tenant 等概念打断。
- 把回答质量和实时输出放进聊天输入框，用户提问时就能直接调整模式，不需要再进设置里找。
- 将消息里的 Assistant/You 改成“法律助手/我”，并去掉头像，让界面更像中文法律问答产品，而不是开发调试聊天框。
- 修复历史版本区域的排版问题，让标题独占一行、三个操作按钮一行等宽排列，避免窄侧栏里出现标题被拆开或按钮掉行的粗糙观感。
- 统一新建、保存和发送等关键按钮颜色，去掉默认组件蓝，让界面看起来是一套完整的产品设计。
- 整理开发者工具面板，把散乱和溢出的控件改成统一尺寸，并移除重复入口，让高级功能保留但不影响普通使用。
- 字体从偏印刷感的衬线字体改成系统无衬线字体，在 Windows 上更清晰，不容易出现发虚和像素颗粒感。
- 风格参考了 kami 的克制和 arena.ai 的清爽应用界面，减少了冷色渐变和装饰噪声。

## 6. 风险与后续优化

- 当前前端主要是展示层产品化，后端接口名和部分内部变量仍保留工程命名，后续可以逐步补充更友好的 API 文档。
- 上传资料入口目前以引导用户粘贴或补充资料为主，后续可以进一步做成更明显的拖拽上传区。
- 回答结构目前主要在前端展示层统一，后续可以让模型提示词更稳定地输出固定法律模块。
- 效果评测仍偏开发者工具，后续可以做一套更面向业务人员的“回答质量说明”。
